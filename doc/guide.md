# Codex 开发经验分析与使用教程（以 RimChat 项目为例）

- Dependencies: `AGENTS.md`, `README.md`, `spec/Index.md`, `build.ps1`
- Responsibility: Explain Codex in plain language for general readers and summarize a practical workflow based on the RimChat project.

## 这篇文档想讲什么

这篇文档写给对 AI 感兴趣、但不是程序员的人。

你不需要懂 RimWorld，也不需要懂这个项目的全部细节。把 RimChat 当成一个真实的 AI 项目案例就可以。

## Codex 是什么

可以把 Codex 理解成一个“能进项目里做事的 AI 助手”。

它和普通聊天 AI 的区别是，它不只是会回答问题，还能：

- 看项目目录和文档
- 搜索代码和配置
- 修改文件
- 运行命令
- 根据报错继续修

但它也不是“全自动程序员”。你还是要告诉它目标、范围和限制。

## 为什么用 RimChat 举例

RimChat 很适合拿来讲 Codex，因为它不是一个只有代码的小项目。它同时包含：

- `RimChat/`：主代码
- `Prompt/`：提示词和模板
- `1.6/Languages/`：多语言文本
- `build.ps1`：构建和部署脚本
- `README.md`、`Api.md`、`config.md`、`spec/Index.md`：项目说明文档

这很像很多真实 AI 项目。你改的往往不只是代码，还会牵动提示词、配置、文档和构建流程。

## GitNexus 是什么

GitNexus 可以简单理解成“项目地图”。

它主要帮你看清这些事情：

- 功能大概分布在哪些模块
- 一个方法会影响谁
- 改这里会不会连带影响别处

如果说 Codex 是“干活的人”，那 GitNexus 更像“导航和雷达”。

最简单的配合方式就是：

1. 先用 GitNexus 看结构和风险。
2. 再让 Codex 去分析或修改。
3. 改完后再检查影响范围。

## 我最想分享的 4 条经验

### 1. 先读后改

这是最重要的一条。

在 RimChat 这种项目里，如果一上来就让 Codex 直接改，很容易只修表面。更稳的做法是先让它读项目结构、相关文档、目标模块和构建脚本。

### 2. 需求要说清楚边界

比起“写很长的提示词”，更重要的是说清楚：

- 要做什么
- 能改哪里
- 不能碰什么
- 怎么算完成

比如这个项目里，常见边界包括：是否兼容旧存档、是否要同步语言键、是否要更新版本日志、是否要跑 `build.ps1`。

### 3. 不要只盯着代码

真实项目里，很多结果不只由代码决定。

在 RimChat 里，`Prompt/`、多语言文件、构建脚本、版本日志都会影响最终效果。所以让 Codex 工作时，最好把这些一起纳入范围。

### 4. 让它顺手做验证

只改文件还不够，最好让它把验证也一起做掉。

一个很实用的要求是：

1. 先读上下文。
2. 再修改。
3. 跑 `build.ps1`。
4. 汇总报错和修复结果。

这样拿到的结果会更接近可用状态。

## 新手怎么开始用

如果你是第一次认真用 Codex，建议这样开始：

### 第一步：先做只读任务

先让它解释模块、整理流程、总结配置，不要急着改代码。

### 第二步：再做小任务

比如补文档、修一个明确的小 bug、补几个语言键。

### 第三步：要求它带验证

不要只要“改好了”，还要它说明改了什么、为什么这么改、怎么验证。

### 第四步：最后再做跨模块任务

等熟悉以后，再让它处理同时涉及代码、提示词、文档和多语言的任务。

## 可以直接拿去用的提问方式

### 1. 只读分析

```text
请先不要改代码。
先结合项目结构和相关文档，用通俗的话解释这个模块是做什么的，
它依赖什么，改它可能会影响什么。
```

### 2. 小功能开发

```text
请先阅读相关模块，再实现这个需求。
目标是……
限制是不要修改……，涉及 UI 文本要同步语言键。
完成后请跑 build.ps1，并告诉我改了什么、怎么验证。
```

### 3. 修 bug

```text
请先只读分析这个问题可能出现在哪些模块，不要急着改。
确认原因后再修复，并说明为什么这样改更稳。
最后跑 build.ps1。
```

## 最后一句话

把 Codex 当成一个很勤快的开发搭子，而不是万能外包。

先让它看懂，再让它动手，效果通常会好很多。
