# FGM Walkthrough - 代码修改过程文档生成器

## 概述

本Skill用于在完成复杂代码修改后，生成结构化的walkthrough.md文档，帮助用户清晰理解修改过程和原理。文档输出到项目的 `/doc` 目录。

## 输出文档结构

生成的walkthrough.md遵循以下结构：

```markdown
Walkthrough: [修改主题标题]

Problem
[清晰描述遇到的问题，包括现象、影响范围]

Root cause
[深入分析问题根本原因，包括技术层面和设计层面的原因]

The Solution: [方案名称]
[描述解决方案的整体思路和策略]

1. [第一层/第一个修改点] — 
[文件名.cs]
[关键代码片段，带行号]
[变更说明]

2. [第二层/第二个修改点] —
[同上格式]

Testing
[测试验证说明，包括测试步骤和预期结果]
```

## 使用流程

### 1. 收集修改信息

在生成文档前，收集以下信息：

- **修改的文件列表** - 所有被修改的文件路径
- **修改前后对比** - 关键代码的变更内容
- **问题背景** - 为什么需要这次修改
- **解决思路** - 如何解决问题的

### 2. 分析代码变更

对每个修改点进行结构化分析：

```
修改点分析模板：
- 文件：完整路径
- 位置：行号范围
- 类型：新增/修改/删除
- 目的：这个修改解决了什么问题
- 原理：为什么这样修改能解决问题
```

### 3. 生成文档

按照模板结构生成walkthrough.md，确保：

- 问题描述清晰具体
- 根因分析深入准确
- 解决方案逻辑完整
- 代码片段带行号和文件链接
- 测试说明可操作

## 文档模板

使用以下模板生成文档：

```markdown
Walkthrough: [主题]

Problem
[问题描述：现象 + 影响]

Root cause: [原因标题]
[技术分析：为什么会出现这个问题]

The Solution: [方案标题]
[方案概述：解决策略和思路]

1. [修改点名称] — 
**文件**: [文件名](file:///绝对路径)

**关键代码**:
```csharp
// 行号 → 代码内容
```

**变更说明**: 
- 新增方法 `MethodName()` — 功能描述
- 修改逻辑 — 变更内容
- 新增规则 — 规则描述

2. [第二个修改点] —
[同上格式]

Testing
Requires in-game testing:
1. [测试步骤1]
2. [测试步骤2]
3. [预期结果]
```

## 输出位置

文档统一输出到项目 `/doc` 目录：

```
项目根目录/
├── doc/
│   ├── walkthrough-[主题]-[日期].md
│   └── ...
```

文件命名格式：`walkthrough-[简短主题]-YYYYMMDD.md`

## 注意事项

1. **代码片段** - 必须带行号，使用 `行号→代码` 格式
2. **文件链接** - 使用 `file:///` 协议的绝对路径
3. **语言** - 与用户对话语言保持一致
4. **简洁性** - 只展示关键代码，避免大段复制
5. **可追溯** - 确保每个修改点都能追溯到具体代码位置
